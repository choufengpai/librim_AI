## 一、背景

当前项目使用 **Qwen2.5-1.5B-Instruct** 作为本地 LLM，基于 MLC-LLM 推理引擎部署。2026年3月2日，阿里巴巴发布了 **Qwen3.5 系列小模型**（0.8B/2B/4B/9B），引发模型选型重新评估的需求。

## 三、架构差异分析

### 3.1 Qwen2.5-1.5B 架构

- **类型**: 标准 Transformer Decoder-only
- **注意力机制**: 全注意力 (Full Attention)
- **计算复杂度**: O(L²) 二次复杂度
- **KV Cache**: 标准 KV Cache，内存占用随序列长度线性增长

### 3.2 Qwen3.5-2B 架构

Qwen3.5 引入全新的混合架构：

#### Gated DeltaNet (75% 层)
- **类型**: 线性注意力机制
- **计算复杂度**: O(L) 线性复杂度
- **内存效率**: 比 KV Cache 更高效
- **参数**:
  - `linear_conv_kernel_dim: 4`
  - Q/K heads: 16, V heads: 16
  - Head dim: 128

#### Gated Full Attention (25% 层)
- **类型**: 标准 GQA 注意力
- **参数**:
  - Q heads: 8, KV heads: 2
  - Head dim: 256
  - RoPE dim: 64
  - Partial rotary factor: 0.25

#### 层结构模式
```
[3 × (Gated DeltaNet + FFN)] → [1 × (Gated Attention + FFN)]
重复 6 次，共 24 层
```

### 3.3 架构优势

| 维度 | Qwen2.5 | Qwen3.5 | 说明 |
|------|---------|---------|------|
| 长上下文性能 | KV Cache 内存爆炸 | 线性内存增长 | 262K 上下文可行 |
| 推理速度 | 标准 | 更快 (线性复杂度) | 长序列优势明显 |
| Thinking Mode | 不支持 | 支持 | 可切换推理深度 |

## 四、推理引擎支持情况

### 4.1 MLC-LLM 支持状态

#### 当前状态
- **Qwen2.5-1.5B**: ✅ 完全支持，有预编译模型
- **Qwen3.5-2B**: ❌ **不支持**

#### 官方 Issue
- GitHub Issue: [mlc-ai/web-llm #778](https://github.com/mlc-ai/web-llm/issues/778)
- 创建时间: 2026-03-02
- 状态: Open
- 反应: 👍 6

#### 阻塞原因
Qwen3.5 的 Gated DeltaNet 架构需要 TVM 编译器新增算子支持：

1. **Gated DeltaNet 算子**: 线性注意力的核心计算
2. **混合层调度**: 3:1 的层交替模式
3. **Multi-Token Prediction (MTP)**: 多 token 预测机制

#### 预估支持时间
- 乐观估计: 2-4 周
- 保守估计: 1-2 个月
- 取决于 MLC-LLM 团队优先级

### 4.2 llama.cpp 支持状态

#### 当前状态
- **Qwen3.5 系列**: ✅ **已支持**
- Commit: [ggml-org/llama.cpp@4248c93](https://github.com/ggml-org/llama.cpp/actions/runs/21860198218)
- GGUF 模型: HuggingFace 已有量化版本

#### Android 部署方式
1. **Termux**: 原生编译运行
2. **NDK 交叉编译**: 生成 Android 原生库
3. **Java/Kotlin 封装**: 需要自行开发 JNI 层

#### 已知问题
- Tool calling 模板问题 (Issue #19872)
- 需要 `--reasoning-parser qwen3` 参数处理 thinking mode

### 4.3 其他框架支持

| 框架 | Qwen3.5 支持 | 说明 |
|------|-------------|------|
| vLLM | ✅ 支持 | 需要 `--reasoning-parser qwen3` |
| SGLang | ✅ 支持 | 需要 `--reasoning-parser qwen3` |
| MLX (Apple Silicon) | ✅ 支持 | mlx-lm (文本), mlx-vlm (视觉) |
| Transformers | ✅ 支持 | `transformers serve` 命令 |

## 五、迁移方案分析

### 5.1 方案一：等待 MLC-LLM 支持

**优点**:
- 保持现有架构不变
- 无需额外开发工作
- MLC-LLM 针对移动端优化

**缺点**:
- 时间不确定 (2周-2月)
- 无法使用新模型能力

**风险评估**: 低风险

### 5.2 方案二：切换到 llama.cpp

**优点**:
- Qwen3.5 已支持
- 社区活跃，更新及时
- GGUF 模型丰富

**缺点**:
- 需要重新开发 Android JNI 封装
- 丢失 MLC-LLM 的移动端优化
- 开发工作量较大 (预估 1-2 周)

**风险评估**: 中等风险

### 5.3 方案三：双引擎支持

**优点**:
- 短期可用 Qwen2.5
- 长期可切换 Qwen3.5
- 灵活性高

**缺点**:
- 架构复杂度增加
- 维护成本上升

**风险评估**: 中等风险

## 六、功能影响分析

### 6.1 当前需求

根据 `TECHNICAL_SOLUTION.md`，核心功能：
- 个人特征识别
- 智能输入增强
- 隐私保护

这些功能主要依赖文本理解能力，**Qwen2.5-1.5B 基本满足需求**。

### 6.2 Qwen3.5 新能力利用

| 新能力 | 潜在应用场景 | 优先级 |
|--------|------------|--------|
| Thinking Mode | 复杂意图理解、多轮对话推理 | 中 |
| 原生视觉 | 截图 OCR、图片内容识别 | 低 |
| 262K 上下文 | 长文本处理、历史记录分析 | 低 |
| 201 语言 | 多语言用户支持 | 低 |

### 6.3 建议

**MVP 阶段**：继续使用 Qwen2.5-1.5B
- 满足核心需求
- 风险可控
- 快速落地

**后续迭代**：根据 MLC-LLM 支持情况决定
- 若 MLC-LLM 在 1 个月内支持：直接迁移
- 若延迟超过 2 个月：评估 llama.cpp 方案
